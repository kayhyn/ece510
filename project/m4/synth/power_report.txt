Power Report -- M4 128-MAC Array
=================================

OpenLane 2 v2.3.10 (containerized via colima/docker)
PDK/SCL: sky130A / sky130_fd_sc_hd
Clock target: 4.0 ns (250 MHz)

Two power estimates are reported, one per top-level taken through OpenLane:

(A) accel_top -- production wrapper, full place-and-route
(B) Single-lane post-PnR power (lane_wrap = mac_array #(.NUM_MAC(1)) with the
    carry-save accumulator), scaled to a 128-lane array estimate

----------------------------------------------------------------------------
(A) accel_top -- post-PnR per-corner power, default switching activity
    Run tag M4_ACCEL  (config_accel.json)
----------------------------------------------------------------------------

Source: runs/M4_ACCEL/.../stapostpnr/<corner>/power.rpt

  Corner             Internal       Switching      Leakage        Total
  nom_tt_025C_1v80   see runs/M4_ACCEL/.../stapostpnr/nom_tt_025C_1v80/power.rpt
  nom_ss_100C_1v60   see runs/M4_ACCEL/.../stapostpnr/nom_ss_100C_1v60/power.rpt
  nom_ff_n40C_1v95   see runs/M4_ACCEL/.../stapostpnr/nom_ff_n40C_1v95/power.rpt

These numbers come from the full `accel_top` PnR (including weight memory,
banked broadcast tree, the 128-lane mac_array, and the result serializer) so
they include clock-tree power on the (banked) broadcast nets and the buffer
chains inserted by timing-repair. Note the `accel_top` PnR uses `L_MAX=64`
(parametric override) so the weight register file is tractable for the
educational sky130 inference flow; a real ASIC would back the 576-deep bank
with SRAM macros, which would lower the weight-storage power.

----------------------------------------------------------------------------
(B) Single placed lane, post-PnR (with CSA), default activity
    Run tag M4_CSA_LANE
----------------------------------------------------------------------------

Source: synth/lane_power_tt.rpt, synth/lane_power_ss.rpt, synth/lane_power_ff.rpt
        (full OpenLane run dir is gitignored due to size).

  Corner             Sequential     Combinational  Clock         Total
  nom_tt_025C_1v80   1.440 mW       0.923 mW       1.229 mW      3.592 mW
  nom_ss_100C_1v60   1.118 mW       0.742 mW       0.977 mW      2.837 mW

  tt split: Sequential 40.1%, Combinational 25.7%, Clock 34.2%.

Naive 128-lane extrapolation (per-lane x 128):

  Corner             Total (single lane)     128x array estimate
  nom_tt_025C_1v80   3.592 mW                ~460 mW
  nom_ss_100C_1v60   2.837 mW                ~363 mW

The 128x scaling is an upper bound: clock-tree power does not strictly scale
linearly under CTS (the H-tree shares high-level buffers), and the shared
control pipeline + the banked broadcast tree are not 128x. The full `accel_top`
post-PnR number in section (A) is the authoritative figure.

Comparison to early pre-PnR estimates:
  Early pre-PnR full mac_array (no CSA, no PnR):  ~255 mW (tt), ~193 mW (ss).
  These earlier numbers were before CTS/buffer insertion and so excluded the
  bulk of clock-tree power; the post-PnR numbers above include it.

Comparison to the M3 single 9-tap lane (M3_SYNTH):
  M3 single lane:   ~1.569 mW total (pre-PnR)
  M4 single lane:   ~3.59 mW total (post-PnR, tt, with CSA)

The increase reflects (a) the added CSA carry register per lane, (b) the
3-stage pipeline registers (M3 was unpipelined), and (c) post-PnR clock-tree
power that the M3 pre-PnR number did not include.

----------------------------------------------------------------------------
Caveats
----------------------------------------------------------------------------

* This is OpenLane default switching activity (alpha 0.5 on inputs). For a
  final energy claim, re-run with a SAIF or VCD annotated from the
  `tb_accel_top` workload (128 active lanes streaming 8 pixels of L=576).
* The slow-corner (ss) numbers include leakage (~1.9 uW total in the lane,
  ~0.24 mW for 128x).
